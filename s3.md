## Simple Storage Service (S3)

Multipart upload allows you to upload a single object to Amazon S3 as a set of parts. Each part is a contiguous portion of the object's data. You can upload these object parts independently, and in any order. For uploads, your updated AWS client automatically calculates a checksum of the object and sends it to Amazon S3 along with the size of the object as a part of the request. If transmission of any part fails, you can retransmit that part without affecting other parts. After all parts of your object are uploaded, Amazon S3 assembles them to create the object. It's a best practice to use multipart upload for objects that are 100 MB or larger instead of uploading them in a single operation.

S3 lifecycle transitions are limited; not all transitions are valid. To be valid a lifecycle transition should adhere to this ordering, where a lifecycle transition is valid if it moves from left to right along this list: S3 Standard -> S3 Standard-IA -> S3 Intelligent Tiering -> S3 One Zone-IA -> S3 Glacier Instant Retrieval -> S3 Glacier Flexible Retrieval -> S3 Glacier Deep Archive.

Versioning-enabled buckets allow us to restore objects from unintended overwriting or deletion. In the case of overwriting an object, we can restore a prior version. On delete, Amazon S3 inserts a delete marker for versioning-enabled buckets' objects, and that becomes the latest version of the object. In this case you can just restore the version prior to the delete marker to restore the object.

By default, an Amazon S3 object is owned by the AWS account that uploaded it.
